# JCT scout — point_in_ring_correct tractability

**Date**: 2026-05-29.
**Scope**: Two-hour targeted search across the now-installed library
ecosystem (fourcolor 1.4.2 + mathcomp-analysis 1.16.0).
**Question**: Is `point_in_ring_correct` (the Phase 3 H1 JCT gap)
tractable given current libraries, or does it remain thesis-scale?

**Verdict**: **RED — confirmed thesis-scale.**

The three search questions all resolve negatively.  Closing
`point_in_ring_correct` from the available libraries requires building
three separable bridges, each ~thesis-scale on its own.  No tractable
multi-session path exists.

## §1 — Search results

### Q1: Fourcolor `Jordan` predicate vs R² geometry

`fourcolor/proof/hypermap.v:263`:

```coq
Definition Moebius_path q :=
  if q isn't x :: p then false else
  [&& uniq q, path clink x p & mem2 p (finv node (last x p)) (node x)].

Definition Jordan := forall q, ~~ Moebius_path q.
```

**Purely combinatorial.**  `q` is a sequence of darts (elements of a
finite type `G : hypermap`); `clink` is the combinatorial closeness
relation `relU (frel (finv node)) (frel face)`; `mem2` checks
sequence membership.

**Zero connection to R² points, R² distance, or geometric interior.**
The Jordan property here says "no Moebius path exists in the
combinatorial hypermap" — a property of the combinatorial
representation, NOT of any embedding into R².

The `realplane.v` topology toolkit (`region`, `connected`, `closure`,
`border`, `adjacent`) is used downstream to STATE the four-color
theorem semantically, but `jordan.v` itself never crosses the
combinatorial-geometric bridge.  fourcolor avoids defining "interior
of a simple closed polygonal curve in R²" entirely — its semantic
content (4CT statement) only uses `region` membership, not Jordan
inside/outside.

**Q1 verdict: RED.**  Fourcolor's Jordan IS NOT the JCT we need.

### Q2: Stdlib R instance of `Real.structure`

`fourcolor/reals/real.v:100`:

```coq
Record structure : Type := Structure {
   val : Type;
   le : rel; sup : set -> val;
   add : val -> val -> val; zero : val; opp : val -> val;
   mul : val -> val -> val; one : val; inv : val -> val
}.
```

`fourcolor/proof/dedekind.v` provides
`Dedekind.real : excluded_middle -> Real.model` — a construction of
the reals via Dedekind cuts on `rat` (mathcomp's rationals).  Uses
ssreflect/mathcomp infrastructure.

**No Stdlib R instance of `Real.structure` exists anywhere in the
installed ecosystem.**  Verified via:

```sh
find ~/.opam -name "*.v" | xargs grep -l "Real.structure" \
  | xargs grep -l "Coq.Reals\|Stdlib.Reals"
# → no matches
```

mathcomp-analysis defines its own real number framework
(`realType` / `realFieldType` typeclasses); it does not implement
fourcolor's `Real.structure`.

**Q2 verdict: RED on bridge availability.**  Building a Stdlib R →
`Real.structure` bridge requires:

- Construct `Structure { val := R; le := Rle; sup := Rsup; add := Rplus;
  ...; inv := Rinv }`.
- Prove all 11 `Real.axioms` (sup_upper_bound, sup_total, add_monotone,
  add_commutative, mul_monotone, etc.) for Stdlib R.

Estimated **2-3 sessions, ~150 lines**.  Not blocking, but a real
cost before any JCT-relevant work can use fourcolor's `realplane.v`.

### Q3: mathcomp-analysis JCT-adjacent results

`mathcomp/analysis/topology_theory/` directory has 22 `.v` files
covering: bool / discrete / metric / order / num / product / quotient
/ separation_axioms / subspace / subtype / supremum / weak topology +
compact, connected, function_spaces.

**`connected.v`** (`mathcomp/analysis/topology_theory/connected.v`)
defines:
- `connected A`
- `separated A B`
- `connected_component`
- `connected_closure`
- `clopen_separatedP`

These are general topological primitives — sufficient for STATING
"interior of a simple closed curve" abstractly via
`connected_component` of the complement, but no JCT theorem connects
the abstract topology to polygons in R².

**`charge.v`** has `jordan_pos` / `jordan_neg` — these are the
**Jordan-Hahn decomposition of signed measures** (jordan_pos = the
positive variation, jordan_neg = the negative variation).  Unrelated
to the Jordan curve theorem; spurious name match.

**No `Jordan_curve` / `simple_closed_curve` / `winding_number` /
`crossing_number` / `point_in_polygon` / `inside_polygon` anywhere
in the installed mathcomp ecosystem.**

**Q3 verdict: RED.**  mathcomp-analysis has the TOPOLOGY VOCABULARY
to state JCT — but not the theorem itself.

## §2 — Gap analysis

### Gap 1: Combinatorial → topological

**Fourcolor's `planar_Jordan` says**: in a planar hypermap, no
Moebius path exists.  Purely combinatorial; no R² content.

**Connection to R² geometry**: `realplane.v` defines `point`,
`region`, `connected`, etc., over `Real.val R` — but `jordan.v` does
NOT bridge `Jordan G` (hypermap) to any region property of
`realplane.v`.  The connection happens DOWNSTREAM in `embed.v` (used
inside the 4CT proof), but that bridge is configuration-specific
(works only for the kernel of a configuration map's perimeter), not a
general "Jordan curve in R² ⇒ separated interior + exterior" result.

**Bridge effort to make `planar_Jordan` usable for `point_in_ring_correct`**:
- Define a hypermap H built from the polygon's ring (one dart per
  ring vertex, three permutations).
- Prove H is planar (`genus H = 0`) — requires Euler-formula
  bookkeeping.
- Prove `Jordan H` via `planar_Jordan`.
- Reformulate `Jordan H` in terms of `realplane.region` (the
  realplane embedding).  **This is the missing bridge** — fourcolor
  avoided this and we'd be inventing it.
- Connect realplane's `connected_component` of R² \ ring to the
  corpus's `point_in_ring` ray-crossing.

**Estimated: ~thesis-scale, 8-15 sessions.**  Each step is non-trivial.

### Gap 2: Real number framework

**Stdlib R instance of `Real.structure`**: missing.  Must build.

**If missing, bridge cost**: 2-3 sessions, ~150 lines.  Construct
the instance, prove the 11 axioms.

### Gap 3: Crossing number ↔ interior

**Any library connects crossing number to interior**: **NO.**

- mathcomp-analysis: no `ray_cast` / `crossing_number` /
  `point_in_polygon` predicates.
- fourcolor: nothing.
- GeoCoq (not installed but searched earlier): nothing.

**This gap requires from-scratch proof.**  The corpus's `point_in_ring`
uses the standard ray-parity characterisation; proving it agrees with
"q is in the bounded component of R² \ |ring|" requires:

- Define the bounded connected component of the complement.
- Prove the ray (from q, axis-aligned, going right) crosses the ring
  an odd number of times iff q is in the bounded component.
- Handle degenerate cases: ray through a ring vertex, ray along an
  edge, ring with self-touching points.

This is essentially the Jordan curve theorem for polygons restated
in terms of ray-parity.  **Substantial, ~thesis-scale on its own.**

### Total bridge effort

```
Gap 1 (combinatorial → R² JCT):     8-15 sessions
Gap 2 (Stdlib R → Real.structure):  2-3 sessions
Gap 3 (crossing-number ↔ interior): thesis-scale
                                    (the JCT itself, basically)
```

**Each gap is roughly independent**; can't easily piggyback one on
another.  Gap 3 alone is the load-bearing piece — even if gaps 1 and
2 close, gap 3 remains the genuine JCT proof.

---

## §3 — Verdict

**RED — confirmed thesis-scale.**

Closing `point_in_ring_correct` from the current library ecosystem
requires building three separable bridges, the third of which (the
crossing-number ↔ interior equivalence) IS essentially the polygonal
JCT and remains thesis-scale.  The other two bridges (Real.structure
instance, fourcolor-to-R² geometric translation) add 10-18 sessions
of infrastructure work on top.

**The available libraries don't shortcut the JCT**:
- Fourcolor proves a COMBINATORIAL Jordan property for planar
  hypermaps, not a GEOMETRIC Jordan property for R² polygons.
- mathcomp-analysis has general topology primitives but not the JCT.

**Recommendation**: keep `point_in_ring_correct` in the deferred
registry as-is (Phase 3 H1).  The library ecosystem doesn't change
the picture.  When/if a published Coq formalisation of JCT for
polygons appears in the ecosystem (current state: nothing), re-open.

Same epistemic position as before the install: Phase 3
`overlay_ng_correct_conditional` remains conditional on H1 (JCT);
`hobby_lemma_4_3_no_proper` (Phase 2) remains in a similar position.
